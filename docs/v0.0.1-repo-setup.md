# v0.0.1 — 仓库管理配置

## 目标

初始化 git 仓库，建立 `dev` / `main` 双分支策略，通过 `post-merge` hook 确保合并到
`main` 时自动过滤开发专用内容，同时不影响将来 clone `main` 的项目正常使用。

---

## 分支策略

| 分支 | 用途 |
|------|------|
| `dev` | 日常开发，包含完整内容 |
| `main` | 干净的模板分支，将来新项目从此 clone |

日常在 `dev` 开发，功能完成后 `git merge dev`（在 `main` 上执行），hook 自动完成过滤。

---

## 各内容的归属

| 内容 | dev | main | clone main 的新项目 |
|------|-----|------|-------------------|
| `packages/` `templates/` `scripts/create.mjs` | ✅ | ✅ | ✅ |
| `docs/` | ✅ | ❌ | ❌ |
| `frontend/` `backend/`（测试产物） | gitignore | ❌ | ✅ 可正常提交 |
| `.githooks/` | ✅ | ✅ | ✅（继承，但不会误触发） |
| `.gitignore.main` | ✅ | ❌（已转为 .gitignore） | ❌ |

---

## .gitignore 两个版本

- `dev` 的 `.gitignore` → 包含 `frontend/` `backend/`（测试产物不提交）
- `main` 的 `.gitignore` → 不包含，让将来的项目自己管理

实现方式：在 `dev` 上维护两个文件：
- `.gitignore` — dev 分支使用
- `.gitignore.main` — hook 触发时复制为 main 的 `.gitignore`，同时作为哨兵文件

---

## Git Hook

### 激活方式

`package.json` 的 `prepare` 脚本在 `pnpm install` 时自动执行：

```json
"prepare": "git config core.hooksPath .githooks && git config branch.main.mergeoptions --no-ff"
```

- `core.hooksPath` — 指向 `.githooks/` 目录激活 hook
- `branch.main.mergeoptions --no-ff` — 强制 main 上的 merge 始终产生 merge commit（`post-merge` 依赖此行为）

**注意**：clone 后必须先运行 `pnpm install` 才能激活 hook。

### `.githooks/post-merge`

触发时机：`git merge` 创建 merge commit 之后立即执行。

> **注**：使用 `post-merge` 而非 `pre-merge-commit`，是因为 git 的 ort 合并策略在调用
> hook 前已确定 commit tree，`pre-merge-commit` 对 index 的修改不会纳入 merge commit。
> `post-merge` 在 merge commit 已创建后触发，通过额外提交一个 filter commit 完成清理。

执行逻辑：

1. 检查当前分支是否为 `main`，不是则直接退出
2. 检查工作区是否存在 `.gitignore.main`（哨兵文件，只在 starter 的 dev 分支存在）
3. 未检测到哨兵 → 直接退出（不影响将来项目自己的 dev → main 合并）
4. 检测到哨兵 → 执行过滤并提交：
   - 将 `.gitignore.main` 内容覆盖 `.gitignore`
   - `git rm -r --cached docs/` + `rm -rf docs/`
   - `git rm --cached .gitignore.main` + `rm -f .gitignore.main`
   - `git commit -m "chore: apply main branch filters"`

结果：每次 merge 产生两个 commit（merge commit + filter commit），main 上的内容始终干净。
