# v0.0.1 — 仓库管理配置

## 目标

初始化 git 仓库，建立 `dev` / `main` 双分支策略，通过 git hook 确保合并到
`main` 时自动过滤开发专用内容，同时不影响将来 clone `main` 的项目正常使用。

---

## 分支策略

| 分支 | 用途 |
|------|------|
| `dev` | 日常开发，包含完整内容 |
| `main` | 干净的模板分支，将来新项目从此 clone |

日常在 `dev` 开发，功能完成后 `git merge dev`（在 `main` 上执行），hook 自动完成过滤。

---

## 各内容的归属

| 内容 | dev | main | clone main 的新项目 |
|------|-----|------|-------------------|
| `packages/` `templates/` `scripts/create.mjs` | ✅ | ✅ | ✅ |
| `.claude/docs/` | ✅ | ✅（来自 docs.main） | ✅（来自 docs.main） |
| `frontend/` `backend/`（测试产物） | gitignore | ❌ | ✅ 可正常提交 |
| `.githooks/` | ✅ | ✅ | ✅（继承，但不会误触发） |
| `.gitignore.main` | ✅ | ❌（已转为 .gitignore） | ❌ |
| `CLAUDE.main.md` | ✅ | ❌（已转为 CLAUDE.md） | ❌ |
| `.claude/docs.main/` | ✅ | ❌（已转为 .claude/docs/） | ❌ |

## 两版本文件

### `.gitignore`

- `dev` 的 `.gitignore` → 包含 `frontend/` `backend/`（测试产物不提交）
- `main` 的 `.gitignore` → 不包含，让将来的项目自己管理

实现方式：在 `dev` 上维护两个文件：
- `.gitignore` — dev 分支使用
- `.gitignore.main` — hook 触发时内容替换到 `.gitignore`，同时作为哨兵文件

### `CLAUDE.md`

- `CLAUDE.md` — dev 分支专用，说明分支策略和 hook 机制
- `CLAUDE.main.md` — hook 触发时内容替换到 `CLAUDE.md`，提供模板项目的架构��明

### `.claude/docs/`

- `.claude/docs/` — dev 内部文档（todo、版本日志）
- `.claude/docs.main/` — hook 触发时替换 `.claude/docs/`，提供模板项目的初始文档

## Git Hook

### 激活方式

`package.json` 的 `prepare` 脚本在 `pnpm install` 时自动执行：

```json
"prepare": "git config core.hooksPath .githooks && git config branch.main.mergeoptions --no-ff && git config merge.use-theirs.name 'Always accept theirs' && git config merge.use-theirs.driver 'cp %B %A'"
```

- `core.hooksPath` — 指向 `.githooks/` 目录激活 hook
- `branch.main.mergeoptions --no-ff` — 强制 main 上的 merge 始终产生 merge commit
- `merge.use-theirs` — 自定义 merge driver，配合 `.gitattributes` 消除哨兵文件冲突

**注意**：clone 后必须先运行 `pnpm install` 才能激活 hook。

### `.gitattributes`

```
.gitignore.main       merge=use-theirs
CLAUDE.main.md        merge=use-theirs
.claude/docs.main/**  merge=use-theirs
```

哨兵文件使用 `use-theirs` merge driver，避免 filter commit 删除后再次 merge 时产生 modify/delete 冲突。

### Hook 架构

git merge 触发的 hook 行为：

| 场景 | 触发的 hook |
|------|------------|
| 干净合并（自动解决） | `pre-merge-commit` |
| 冲突解决后手动 commit | `pre-commit` |

因此使用两个 hook 共享同一套过滤逻辑：

- **`.githooks/pre-merge-commit`** — 从 `GITHEAD_*` 环境变量获取 MERGE_HEAD SHA，调用共享脚本
- **`.githooks/pre-commit`** — 检测 `.git/MERGE_HEAD` 文件，调用共享脚本
- **`.githooks/apply-main-filters`** — 共享过滤逻辑，接收 MERGE_HEAD SHA 作为参数

### 过滤逻辑（`apply-main-filters`）

1. 检查当前分支是否为 `main`，不是则直接退出
2. 检查 MERGE_HEAD 树中是否存在 `.gitignore.main`（哨兵文件）
3. 未检测到哨兵 → 直接退出（不影响将来项目自己的 dev → main 合并）
4. 检测到哨兵 → 执行过滤（所有内容从 MERGE_HEAD 树中读取，不依赖工作区）：
   - 将 `.gitignore.main` 内容覆盖 `.gitignore`，删除 `.gitignore.main`
   - 将 `CLAUDE.main.md` 内容覆盖 `CLAUDE.md`，删除 `CLAUDE.main.md`
   - 删除 `.claude/docs/`，将 `.claude/docs.main/` 重命名为 `.claude/docs/`

结果：每次 merge 产生一个 merge commit（包含过滤后的内容），main 始终干净。
